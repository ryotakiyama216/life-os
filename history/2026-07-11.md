# 2026-07-11 Life OS 開発記録

## 今日やったこと概要
要件ヒアリングを行い、MVP（見た目＋localStorage版）を一気通貫で構築した。Next.js 14 + shadcn/ui + Zustand(localStorage永続化) で、GTD方式のTodo管理ツールの土台が動く状態になった。

## ヒアリングで決まったこと

### プロダクトの狙い
- 目標(Goal) → プロジェクト(Project) → タスク(Task) → 今日のTodo、の階層で管理。ただしプロジェクトを経由しない目標や、習慣だけで達成する目標もあり得る前提。
- Notionは多機能すぎて一人運用だと使わなくなる、という課題感があるため「シンプルさ」を最優先。
- GTD（Getting Things Done）の考え方を採用。緊急タスクや目標と無関係な予定も扱えるようにする。
- 「今、目標達成のために何をすべきか」だけを考えられる状態を目指す。期限切れタスクの自動検出・誘導、日付が今日のタスクの自動表示など、Notionで自分では設定しきれなかった部分を最初から仕組み化する。
- 将来的にSNS投稿・伸びの分析機能も追加したい（今回は着手せず）。

### 技術・認証・AI
- フロントエンド: Next.js、バックエンド/DB: Supabase、ホスティング: Vercel。
- 認証は自分専用（1人運用）でよい。将来Supabase Authを使う想定だが今回は未実装。
- AI機能（チャット/アシスタント、目標→プロジェクト→タスクの自動棚卸し、毎日/毎週のレビュー生成）はやりたいが、**今回のMVPには含めない**。Claude課金を活用してツール内AIを実装したいという要望あり（次フェーズ）。

### 今回のセッションのスコープ判断
当初はTodayビューまでフル実装する想定だったが、ヒアリングの結果「まず見た目とローカルストレージでの動作確認を優先し、DB接続は次回以降」という方針に変更。これにより認証・Supabase構築を待たずにUI/UXを検証できるようにした。

### データ設計の方針
- カラムは固定（Notionのような自由追加式プロパティではない）。ただし将来のSupabaseスキーマにそのまま移行できる型設計にした。
- 優先度は P1〜P4（GTD/Notion的な4段階）。
- 朝のスケジュールは「時間割ブロック形式」（時刻＋タイトルのブロックを追加・並び替えできる構造化データ）。

## 実装した内容

### 技術スタック（今回選定）
- Next.js **14系**（App Router, TypeScript, src/ディレクトリ構成）
  - 理由: ローカルのNode.jsが v18.17.0 / v18.9.0 系で、Next.js 15以降が要求する Node 18.18+ / 20+ を満たしていなかったため。将来Node 20にアップグレードした際にNext.js最新版へ移行可能。
- Tailwind CSS + shadcn/ui（style: new-york, baseColor: zinc＝モノトーン）
  - shadcn CLIの最新版（4.x系）もNode 20+必須で動かなかったため、`shadcn@2.3.0` を使用。
  - `tailwind.config.ts` のカラー定義が `hsl(var(--x))` 形式だったが、globals.cssの生成値が `oklch(...)` 関数そのものだったため不整合が発生 → `var(--x)` 直接参照に修正済み（重要な既知バグの回避、次回shadcn関連の変更をする際は要注意）。
- 状態管理・永続化: Zustand + `persist`（localStorage、キー: `life-os-store`）。将来Supabase化する際はstoreのCRUDメソッドの中身だけ差し替えればよい設計。
- Markdown: `react-markdown` + `remark-gfm` + `@tailwindcss/typography`（`prose`クラス）。編集はシンプルなtextarea＋プレビュー切り替え（重量級エディタは導入せず）。
- テーマ: `next-themes`（ライト/ダーク、モノトーン基調を両対応）。

### データモデル（`src/types/index.ts`）
Goal / Project / Task / Habit / HabitLog / InboxItem / Note / MorningBlock を定義。将来のSupabaseテーブル設計にほぼそのまま流用できる想定。

### 主要機能
- **Today (`/`)**: 期限切れタスクの警告表示＋今日やる/明日に延期/削除などのアクション、今日期限・今日予定のタスク一覧、「今やるべき候補」（目標/プロジェクトの優先度から逆算したフォーカス提案 = `src/lib/priority.ts`）、今日の習慣チェックリスト、朝の時間割ブロック。
- **Inbox (`/inbox`)**: クイックキャプチャで書き溜め、各アイテムをタスク/プロジェクト/目標/習慣/メモ/いつかリストに仕分け。
- **目標 (`/goals`, `/goals/[id]`)**: 状態・優先度でのフィルタ/ソート、詳細ページでMarkdown説明＋紐づくプロジェクト/タスクの管理。
- **プロジェクト (`/projects`, `/projects/[id]`)**: 同様にフィルタ/ソート＋タスク一覧。
- **タスク (`/tasks`)**: 状態・優先度・目標・プロジェクト・キーワードでの絞り込み、期限日/優先度/状態/作成日での並び替えができるテーブルビュー。
- **習慣 (`/habits`)**: 頻度（毎日/曜日指定）と実施時刻の設定。Todayページに自動反映。
- **ページ・メモ (`/notes`, `/notes/[id]`)**: Markdownで自由に書けるメモ。目標/プロジェクトへの関連付け、タグ付け可能。

### 優先度エンジン（`src/lib/priority.ts`）
1. 期限切れを最優先表示（超過日数で強調度アップ）
2. 今日期限・今日予定のタスク
3. 日付未設定でも、紐づく目標・プロジェクトの優先度(P1〜P4)を合成したスコアで「今やるべき候補」を提示 → 好きなタスクに流されず、目標から逆算した優先順位を機械的に提示する仕組み。

## 追記: AI機能「目標→プロジェクト/タスクの自動棚卸し」を追加

### ヒアリングで確認したこと
- **Claude.aiのチャット課金とAnthropic APIは別課金**であることを説明。ユーザーはまだAPIキー未発行だったため、console.anthropic.comでの発行手順を案内。
- 最初に実装するAI機能は「目標→プロジェクト/タスクの自動棚卸し」（3案のうち最優先として選択）。
- モデルはコストと品質のバランスから **Claude Sonnet 5**（`claude-sonnet-5`）を選定。個人ツールで頻繁に呼ぶ想定のため、Opus 4.8（単価5倍）は見送り。

### 実装内容
- `@anthropic-ai/sdk` と `zod` を追加。
- `src/lib/anthropic.ts`: サーバー専用のAnthropicクライアント（`ANTHROPIC_API_KEY`が未設定なら例外）。
- `src/app/api/ai/breakdown/route.ts`: POSTエンドポイント。目標のタイトル・説明を受け取り、`client.messages.parse()` + `zodOutputFormat`で構造化出力（プロジェクト案＋直接タスク案、それぞれ優先度P1〜P4付き）を返す。APIキー未設定時は日本語でわかりやすいエラーを返す。
- `src/components/goal/ai-breakdown-dialog.tsx`: 目標詳細ページに置く「AIで棚卸しする」ボタン。ダイアログを開くと自動でAI呼び出し→提案されたプロジェクト/タスクをチェックボックスで選択→選んだものだけをZustand storeに追加。
- `src/app/goals/[id]/page.tsx` にボタンを配置。
- `.env.local.example` を追加（`ANTHROPIC_API_KEY=`）。README.md・CLAUDE.mdにAPIキー取得手順とAI実装方針を追記。

### 動作確認
- `npm run build` 成功（`/api/ai/breakdown` ルートが認識されている）。
- APIキー未設定の状態で `POST /api/ai/breakdown` を叩き、想定通り日本語のエラーメッセージ（500）が返ることを確認。
- **実際にAPIキーを設定した状態でのAI応答の中身（提案内容の質）はまだ確認していない** — ユーザーがAPIキーを設定次第、実機で試してほしい。
